## Completed Design Decisions

STAG sync now follows a Git-like object identity model.

- Graph record IDs are opaque and collision-resistant.
- The same graph record ID is used in local and remote stores.
- Sync pushes and pulls records without local/global ID translation.
- `idmap.jsonl`, `shared_id`, and `global_id` are no longer part of the sync
  identity design.
- Remote records identify each graph record by the same ID already present in
  the record body.

Git analogy:

- Git commits keep the same hash locally and remotely.
- STAG graph records similarly keep the same opaque ID locally and remotely.
- UUID-backed opaque IDs are sufficient for this project.

## Current Implementation

Core files:

- `src/stag/core/ids.py`
  - Defines `opaque_id(prefix)`.
- `src/stag/core/run.py`
  - `RunHandle._next_id(prefix)` returns opaque IDs.
  - Root node is opaque and stored via graph metadata.
- `src/stag/cli/commands/init.py`
  - `run_init_command()` returns `{"run_id": ..., "root_node_id": ...}`.
- `src/stag/core/sync/local.py`
  - File-backed sync push/pull implementation.
  - No longer reads or writes `idmap.jsonl`.
  - Push writes `record_id`, not `shared_id`.
  - Pull applies remote bodies directly using their existing opaque IDs.
  - Batch IDs are generated by `new_sync_id("batch")`.
- `src/stag/core/sync/records.py`
  - `flatten_batches()` exposes `record_id`.
  - `record_id_for_body()` extracts the record ID from the body.
  - `new_sync_id()` is only for sync-layer envelope IDs such as `batch_id`.
- `src/stag/core/sync/shared_store.py`
  - Keeps the shared store boundary.
- `src/stag/cli/commands/sync.py`
  - CLI wrapper for sync init/status/push/pull.
- `tests/cli/test_sync.py`
  - Asserts remote records have `record_id`.
  - Asserts remote records do not have `shared_id`.
  - Keeps the local A -> shared -> local B round trip.

Deleted:

- `src/stag/core/sync/idmap.py`

Remote record shape:

```json
{
  "record_kind": "node",
  "record_id": "n_<uuid>",
  "body": {
    "node_id": "n_<uuid>",
    "metadata": {}
  }
}
```

Kind mappings:

- node: `record_id == body["node_id"]`
- input_transition: `record_id == body["input_transition_id"]`
- output_transition: `record_id == body["output_transition_id"]`
- payload: `record_id == body["payload_id"]`
- view: `record_id == body["view_id"]`

Batch envelope shape:

```json
{
  "seq": 1,
  "batch_id": "batch_<uuid>",
  "operation": "anchor",
  "records": [ ... ],
  "actor": { ... },
  "origin": { ... },
  "created_at": "..."
}
```

## Important Constraints

- Do not reintroduce sequential graph record IDs.
- Do not assume `n_0000`, `it_0001`, etc.
- Do not add a compatibility shim for old sequential IDs unless explicitly
  requested.
- Do not reintroduce local/global ID translation.
- Do not reintroduce `idmap.jsonl`.
- Do not reintroduce `shared_id` for graph records.
- Keep sync file-backed and local-only for now.
- Do not add HTTP/Postgres yet.
- Keep `SharedRunStore` as the backend boundary.
- Prefer small commits.

## Recommended Next Work

The sync identity cleanup is done. The next useful work is to harden sync
semantics around concurrent local/shared DAG evolution.

Suggested next task:

1. Review pull/push conflict behavior.
2. Define what happens when two local workspaces push different records that
   extend the same node.
3. Define whether this is always allowed as DAG branching, or whether some
   operations require conflict detection.
4. Add tests for at least these cases:
   - Two workspaces independently add different transitions from the same node.
   - One workspace pulls after another workspace pushes.
   - Re-pushing the same local records is idempotent.
   - A remote record with the same `record_id` but different body is rejected.
5. Keep the file-backed shared store; do not introduce a network service yet.
6. Run `PYTHONPATH=src pytest`.
7. Commit focused changes.

Possible implementation direction:

- Treat multiple transitions from the same node as normal DAG branching.
- Treat same `record_id` with different body as a hard conflict.
- Continue using body-derived keys only for detecting whether an equivalent
  record already exists remotely.
- Keep record identity stable across local and shared stores.
